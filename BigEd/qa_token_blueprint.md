> Full tracking of API token consumption, cost calculation, and delta comparison across runs, versions, and time periods.

## Section 1: Token Usage Tracking

### How `resp.usage` Works

Every Anthropic API response includes a `usage` object with these fields:

| Field | Type | Meaning |
|-------|------|---------|
| `input_tokens` | int | Total input tokens (includes cached) |
| `output_tokens` | int | Tokens generated by the model |
| `cache_read_input_tokens` | int | Tokens served from prompt cache (free/discounted) |
| `cache_creation_input_tokens` | int | Tokens written to cache on this call (billed at 1.25x) |

**Cache vs non-cache breakdown:**
- `fresh_input_tokens = input_tokens - cache_read_input_tokens - cache_creation_input_tokens`
- Cache reads are billed at 10% of base input rate (90% savings)
- Cache creation is billed at 125% of base input rate (one-time cost, amortized over cache TTL)
- Output tokens are always billed at the full output rate regardless of caching

### Per-Call Logging

Every `call_complex()` invocation should capture:
- **skill**: which skill dispatched the call (from task context)
- **model**: resolved model ID (e.g. `claude-sonnet-4-6`)
- **input_tokens / output_tokens**: from `resp.usage`
- **cache_read_tokens / cache_create_tokens**: from `resp.usage`
- **cost_usd**: calculated from model pricing (see Section 2)
- **timestamp**: UTC datetime of the call
- **task_id**: linked task from `tasks` table (nullable for ad-hoc calls)
- **agent**: worker name that made the call

### DB Schema: `usage` Table

```sql
CREATE TABLE IF NOT EXISTS usage (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    skill       TEXT NOT NULL,
    model       TEXT NOT NULL,
    input_tokens      INTEGER NOT NULL DEFAULT 0,
    output_tokens     INTEGER NOT NULL DEFAULT 0,
    cache_read_tokens INTEGER NOT NULL DEFAULT 0,
    cache_create_tokens INTEGER NOT NULL DEFAULT 0,
    cost_usd    REAL NOT NULL DEFAULT 0.0,
    task_id     INTEGER,
    agent       TEXT
);
CREATE INDEX IF NOT EXISTS idx_usage_skill ON usage(skill);
CREATE INDEX IF NOT EXISTS idx_usage_created ON usage(created_at);
```

## Section 2: Cost Calculation

### Model Pricing Tiers (per million tokens)

| Model | Input | Output | Cache Read | Cache Create | Batch Input |
|-------|-------|--------|------------|--------------|-------------|
| claude-haiku-4-5 | $0.80 | $4.00 | $0.08 | $1.00 | $0.40 |
| claude-sonnet-4-6 | $3.00 | $15.00 | $0.30 | $3.75 | $1.50 |
| claude-opus-4-6 | $15.00 | $75.00 | $1.50 | $18.75 | $7.50 |

### Cost Formula

```python
def calculate_cost(usage, model_id: str) -> float:
    """Calculate USD cost from usage object and model pricing."""
    rates = PRICING[model_id]  # lookup from table above

    fresh_input = usage.input_tokens - getattr(usage, "cache_read_input_tokens", 0) - getattr(usage, "cache_creation_input_tokens", 0)

    cost = (
        fresh_input * rates["input"] / 1_000_000
        + usage.output_tokens * rates["output"] / 1_000_000
        + getattr(usage, "cache_read_input_tokens", 0) * rates["cache_read"] / 1_000_000
        + getattr(usage, "cache_creation_input_tokens", 0) * rates["cache_create"] / 1_000_000
    )
    return round(cost, 6)
```

### Cache Savings Formula

```
savings_usd = cache_read_tokens * (full_input_rate - cache_read_rate) / 1_000_000
```

For Sonnet: each 1M cache-read tokens saves `$3.00 - $0.30 = $2.70`.

### Batch Discount

Message Batches API provides 50% savings on input tokens for non-real-time workloads. Applicable skills: `plan_workload`, `review_discards`, `synthesize`, `code_review` (batch mode).

### Per-Skill Cost Estimation

Aggregate from historical `usage` rows:

```sql
SELECT skill,
       COUNT(*) as calls,
       SUM(cost_usd) as total_cost,
       AVG(cost_usd) as avg_cost_per_call,
       SUM(input_tokens) as total_input,
       SUM(output_tokens) as total_output
FROM usage
WHERE created_at >= date('now', '-30 days')
GROUP BY skill
ORDER BY total_cost DESC;
```

## Section 3: Delta Comparison Methodology

### Run-over-Run

Same skill, same (or similar) input → compare token counts to detect regressions.

**Use case:** After modifying a skill's system prompt, verify token usage didn't spike.

```python
def compare_runs(skill: str, run_a_id: int, run_b_id: int) -> dict:
    """Compare two individual runs of the same skill."""
    a = db.get_usage(run_a_id)
    b = db.get_usage(run_b_id)
    return {
        "metric": "input_tokens",
        "previous": a["input_tokens"],
        "current": b["input_tokens"],
        "delta_pct": round((b["input_tokens"] - a["input_tokens"]) / a["input_tokens"] * 100, 1),
        "direction": "up" if b["input_tokens"] > a["input_tokens"] else "down",
    }
```

### Version-over-Version

Before/after code changes → did prompt efficiency improve?

Link usage records to git commits or version strings via a `version` column or by correlating timestamps with deploy events.

```sql
-- Compare average token usage between two date ranges (proxy for versions)
SELECT skill,
       AVG(CASE WHEN created_at BETWEEN :v1_start AND :v1_end THEN input_tokens END) as v1_avg,
       AVG(CASE WHEN created_at BETWEEN :v2_start AND :v2_end THEN input_tokens END) as v2_avg
FROM usage
GROUP BY skill
HAVING v1_avg IS NOT NULL AND v2_avg IS NOT NULL;
```

### Time-Series

Daily/weekly aggregates → trend analysis, anomaly detection.

```sql
SELECT date(created_at) as day,
       SUM(cost_usd) as daily_cost,
       SUM(input_tokens) as daily_input,
       SUM(output_tokens) as daily_output,
       SUM(cache_read_tokens) as daily_cache_hits
FROM usage
GROUP BY date(created_at)
ORDER BY day DESC
LIMIT 30;
```

Anomaly detection: flag any day where `daily_cost > 2 * avg(daily_cost over last 14 days)`.

### Skill Ranking

Sort skills by cost-per-call to identify optimization targets:

```sql
SELECT skill,
       COUNT(*) as calls,
       AVG(cost_usd) as avg_cost,
       AVG(input_tokens) as avg_input,
       AVG(output_tokens) as avg_output,
       AVG(cache_read_tokens * 1.0 / NULLIF(input_tokens, 0)) as cache_hit_rate
FROM usage
GROUP BY skill
ORDER BY avg_cost DESC;
```

### Delta Format

All comparison outputs use a consistent structure:

```json
{
    "metric": "input_tokens",
    "previous": 1250,
    "current": 980,
    "delta_pct": -21.6,
    "direction": "down"
}
```

Direction values: `"up"` (increase), `"down"` (decrease), `"flat"` (< 1% change).

## Section 4: Efficient Token Patterns

### Ephemeral Cache Placement

Cache breakpoints should be placed at **stable prefix boundaries** — content that doesn't change between calls:

1. **System prompt** — already cached via `cache_control: {"type": "ephemeral"}` in `_call_claude()`
2. **Tool definitions** — if using tool_use, cache the tools array
3. **Context documents** — RAG-retrieved chunks that are reused across a conversation
4. **Few-shot examples** — static examples in the prompt

Order matters: place the most stable content first, cache breakpoint after it, then variable content.

### Prompt Compression

**When to summarize vs pass full context:**
- **Full context** when: task requires precise details (code review, security audit), input < 4K tokens
- **Summarize** when: context is background/reference only, input > 8K tokens, repeated similar content
- **Truncate** when: only recent history matters (last N turns), content has clear recency ordering

**Compression techniques:**
- Strip markdown formatting from RAG chunks (saves ~15-20% tokens)
- Deduplicate overlapping RAG results before injecting
- Use structured output (JSON) over prose for intermediate results

### Model Routing

Match model capability to task complexity (from `CLAUDE.md`):

| Model | Use When | Token Cost Ratio |
|-------|----------|-----------------|
| `claude-haiku-4-5` | Classification, routing, simple extraction, yes/no decisions | 1x (baseline) |
| `claude-sonnet-4-6` | Generation, analysis, code review, most skill work | 3.75x |
| `claude-opus-4-6` | Complex reasoning, multi-step planning, architecture decisions | 18.75x |

**Rule of thumb:** Start with Haiku. If output quality is insufficient, step up. Most fleet skills should run on Sonnet; only `lead_research`, `plan_workload`, and complex `code_review` need Opus consideration.

### Batch Mode

Use Message Batches API (50% input savings) for non-real-time workloads:
- `plan_workload` — bulk task decomposition
- `review_discards` — batch quality filtering
- `synthesize` — multi-source aggregation
- `code_review` — batch mode for multiple files

Already implemented: `call_complex_batch()` and `check_complex_batch()` in `_models.py`.

### Context Window Management

**Truncation strategies:**
- **Hard truncation:** Cut at token limit, add "[truncated]" marker. Simple but lossy.
- **Sliding window:** Keep first N tokens (system context) + last M tokens (recent). Good for conversations.
- **Semantic truncation:** Use embeddings to keep most relevant chunks. Best quality, highest overhead.

**Recommendation for fleet skills:**
- System prompt: always full (cached, so effectively free after first call)
- User content: hard truncate at 80% of remaining context budget
- Reserve 20% of context for output generation

### `db.py` Changes

**File:** `fleet/db.py`

Add `usage` table creation to schema init, plus two functions:

```python
def log_usage(skill, model, input_tokens, output_tokens,
              cache_read_tokens=0, cache_create_tokens=0,
              cost_usd=0.0, task_id=None, agent=None):
    """Insert a usage record after each API call."""
    conn = _get_conn()
    conn.execute(
        """INSERT INTO usage (skill, model, input_tokens, output_tokens,
           cache_read_tokens, cache_create_tokens, cost_usd, task_id, agent)
           VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?)""",
        (skill, model, input_tokens, output_tokens,
         cache_read_tokens, cache_create_tokens, cost_usd, task_id, agent),
    )
    conn.commit()


def get_usage_summary(period="day", group_by="skill"):
    """Aggregate usage data by period and grouping."""
    period_map = {"day": "-1 day", "week": "-7 days", "month": "-30 days"}
    since = period_map.get(period, "-7 days")
    conn = _get_conn()
    rows = conn.execute(f"""
        SELECT {group_by},
               COUNT(*) as calls,
               SUM(input_tokens) as total_input,
               SUM(output_tokens) as total_output,
               SUM(cache_read_tokens) as total_cache_reads,
               SUM(cost_usd) as total_cost
        FROM usage
        WHERE created_at >= datetime('now', ?)
        GROUP BY {group_by}
        ORDER BY total_cost DESC
    """, (since,)).fetchall()
    return [dict(r) for r in rows]
```
